# A brain-analogous AI memory system, dreaming on a $700 laptop's NPU at 15W

**2026-04-21** — Andre + Hermes

We built, proved, and published the first (known-to-us) working pipeline that runs a full LLM-powered cognitive memory system — with offline dream-consolidation cycles — entirely on an AMD Ryzen AI laptop's XDNA NPU under Linux. No cloud calls. No GPU. ~15W sustained power. End-to-end coherent output generated by the hardware in this very sentence.

## The stack

```
  Your memories            CerebroCortex                  ryzenai-serve              Krackan Point NPU
  ─────────────            ─────────────                  ─────────────              ─────────────────
  episodic      ───┐                                                                 XDNA2 aie2p 6x8
  semantic      ───┼──► Dream Engine ──► LLMClient ──HTTP──► FastAPI ──► onnxruntime-genai
  procedural    ───┤    (6-phase        (OpenAI-compat)     (/v1/chat/completions)    Llama-3.2-3B-16K
  affective     ───┤     consolidation                                                ~15W sustained
  prospective   ───┤     cycle)                                                       10.5 GB RAM peak
  schematic     ───┘                                                                  11 tok/s decode @ 16K ctx
```

Three independent pieces of infrastructure, bolted together by an OpenAI-compatible HTTP interface. Each can be used on its own, or swapped out.

## The three repos

### 1. [ryzenai-lab](https://github.com/buckster123/ryzenai-lab)

Benchmark suite + results for AMD Ryzen AI NPU on Linux — 12 models in NPU-4K, 5 models in NPU-16K, full context-scaling sweeps on Krackan Point. First public Krackan numbers we've found online, and they **beat AMD's published Strix numbers** on Llama-3.2-3B (985/25 t/s prefill/decode vs AMD's 865/17.6). Empirical foundation for the rest of the stack.

### 2. [ryzenai-serve](https://github.com/buckster123/ryzenai-serve) (this repo)

OpenAI-compatible HTTP server wrapping `onnxruntime-genai` for AMD XDNA NPU models on Linux. Fills a genuine gap — AMD ships `model_benchmark` (C++) and `run_model.py` (interactive Python) but no HTTP server, so the NPU can't be used behind LangChain, LMStudio clients, or any OpenAI-compatible SDK. Now it can. FastAPI, streaming SSE, jinja2 chat templates, single-model load-at-startup, ~300 lines of code.

### 3. [CerebroCortex](https://github.com/buckster123/CerebroCortex)

Brain-analogous AI memory system by Andre (started well before this project). Six memory modalities (episodic, semantic, procedural, affective, prospective, schematic), ACT-R power-law activation + FSRS retrievability + vector similarity + salience (4-weight combined scoring), associative network with typed/weighted links and spreading activation, and — crucially — a **Dream Engine** that runs 6-phase consolidation cycles (SWS replay, pattern extraction, schema formation, emotional reprocessing, pruning, REM recombination) using an LLM. Designed for Raspberry Pi 5 clusters but portable anywhere.

## The zero-code moment

CerebroCortex already shipped an `OpenAICompatProvider` with automatic failover. That meant the entire NPU integration was *pure configuration*:

```json
{
  "llm": {
    "primary_provider": "openai_compat",
    "primary_model":    "Llama-3.2-3B-Instruct-npu-16K",
    "openai_compat_base_url": "http://127.0.0.1:8000",
    "no_think": false,
    "strip_think": false
  }
}
```

`settings.json` placed in `~/.cerebro-cortex/`. No CerebroCortex code changed. The brain-memory system now dreams on an NPU.

(Well — one CLI bugfix. CerebroCortex's CLI entry point wasn't calling `load_on_startup()` like its MCP server was. Seven lines to the upstream repo and it's permanent.)

```
  Success:      True

  Phases:
    sws_replay             0.0s    No unconsolidated episodes
    pattern_extraction    14.0s    Extracted 3 procedures from 1 cluster
    schema_formation       0.0s
    emotional_reprocessing 0.0s
    pruning                0.0s    Decayed 10 memories
    rem_recombination     13.8s    Checked 3 pairs, created 3 new connections
```

The dream engine, running entirely on NPU-served Llama-3.2-3B, looked at our memories and extracted three new procedural ones with correct `derived_from` provenance links — including this one, which generalized a specific failure into a pattern:

> *"When compiling models, check the SDK version compatibility, especially for Qwen3.5"*

This derived_from a semantic memory about Qwen3.5's hybrid linear-attention/SSM/MTP architecture being incompatible with onnxruntime-genai's dispatcher as of SDK 1.7.1. The consolidation engine took a specific incident and abstracted it. That's the whole point.

## The surprise bug

The bench sweep picked `Qwen2.5-3B-Instruct` as the obvious winner — 21.8 t/s decode vs Llama-3.2-3B's 18.2, 7.3 GB peak RAM vs 10.5, 2.6s init vs 3.8. Smaller, faster, lighter. We pinned it as the default, wrote it into docs, updated memories.

Then we served it through the Python API and got:

```
"TheGGGGGGGGGGGGGGGGGGGGGGGGGG..."
```

First 1-2 tokens correct, then collapse to a single repeating token. AMD's own `run_model.py` reproduces it with a telling warning: `Unsupported model type 'qwen2'. Defaulting to ONNXTokenizerWrapper`. The `model_benchmark` C++ binary runs Qwen2.5 cleanly (hence the bench numbers), but every Python-based serving path breaks. Same failure on Qwen2.5-7B, Phi-4-mini-instruct, Phi-4-mini-reasoning. Only Llama-3.2-3B survives.

Lesson written into the skill files for the next session and every future one: **bench ≠ serving**. Validate Python output quality before trusting bench numbers for a serving use case.

Documented in `ISSUES.md`. Open question — waiting on AMD's next SDK drop.

## What this enables

- **Overnight consolidation**, cron-scheduled, ~15W sustained, while the user sleeps
- **Zero cloud dependency** for memory maintenance (no Claude-API bill, no latency, no privacy concern)
- **Standard OpenAI-compatible API** so the NPU is a drop-in backend for LangChain, LMStudio clients, any OpenAI SDK
- **Proof-of-concept for local-first Apexaurum**, where compute lives on user laptops and workstations and the cognitive stack runs locally

## What's left

- **BGE embedder on NPU** — swap CC's sentence-transformers (CPU) for Vitis AI-compiled BGE. Last CPU bottleneck in the stack.
- **Native CC `NPUProvider` class** — smart defaults, model auto-discovery via `/v1/models`, cut CC v0.3.0 with "works out of the box on Ryzen AI" story.
- **Cloud compile research** — Quark-quantize Qwen3 dense (8B/14B) on vast.ai, compile via AMD's `model_generate`, first community-compiled Krackan Qwen3 model.
- **The Qwen2.5/Phi-4 fix** — either track AMD's SDK releases or rebuild the models with untied embeddings to see if that's the actual cause.

## Hardware

**Lenovo 83HX laptop, AMD Ryzen AI 5 340 "Krackan Point"**, 24 GB LPDDR5, integrated Radeon 840M, Ubuntu 25.10, kernel 6.19.13. NPU is XDNA2 `aie2p 6x8` (48 AIE tiles — Strix has 64). Out-of-tree `amdxdna` driver from AMD's GitHub (fixes SVA -95 regression), XRT 2.23, Ryzen AI SDK 1.7.1. pmode=turbo (the undocumented setting that ~doubles decode perf — see the `amd-xdna-linux` skill).
